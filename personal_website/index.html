Here there is the exaplnation of all these key-values in the JSON:

  • CallerReference: This is a unique identifier for the distribution. This can be any string, it is used to identify the distribution request in the AWS Cloudfront servers.
  • Aliases: This is a list of CNAMES that you want to associate with the distribution. In this case, there is only 1 CNAME that the distribution will be associated with. This is my domain "jaxelc.cetystijuana.com".
  • DefaultRootObject: The object that CloudFront returns when a viewer requests the root URL of your distribution. In this case, it is "index.html" which is the main html file that renders the website.
  • Origins: The origin information for the distribution. In this case, there is only 1 origin, which is the S3 bucket. So here need to define the information for the origin. The Id is an unique identifier for your origin, this is unique within the distribution. The DomainName is the origin DNS that was generated when the S3 bucket was created. The CustomHeaders is a list of HTTP header names and values that CloudFront adds to the requests that it sends to the origin. The CustomOriginConfig is a type to specify an origin that is not an Amazon S3 bucket, because this we are using an Amazon S3 bucket to host a static website we need to use this type with some settings:
    • HTTPPort: This is the HTTP port that CloudFront uses to connect to the origin. It is specified to port 80, so this is the port that the origin listens to.
    • HTTPSPort: This is the HTTPS port that CloudFront uses to connect to the origin. It is specified to port 443 that is the HTTPS port that the origin listens to.
    • OriginProtocolPolicy: This specifies the HTTP or HTTPS protocols that CloudFront uses to connect to the origin. It is set to "http-only" to specify the protocol that CloudFront should use when communicating with the origin, so this means that CloudFront will only use HTTP to communicate with the origin, and all request from viewers to the CloudFront distribution will be automatically redirected to HTTPS.
    • OriginSslProtocols: Specifies the minimun SSL/TLS protocol that CloudFront uses when connecting to the origin over HTTPS. Here we only specify 3 protocols, TLSv1, TLSv1.1 and TLSv1.2. For more information on minimun origin SSL protocol, visit Minimun Origin SSL Protocol.
  • DefaultCacheBehavior: Here we specify the cache settings for the distribution. This is a complex type because describes the default cache for the origin and there must be only one default cache behavior. Need to declare some settings for this behavior:
    • TargetOriginId: Here we specify what origin id we want to apply the cache settings to. In this case this has to be same as the origin id specified before, in this case it is "jaxelc.cetystijuana.com".
    • ViewerProtocolPolicy: This is the protocol that viewers can use to access the files in the origin specified by the TargetOriginId when a request matches the path pattern of the origin cache behavior path pattern. This pattern can be ommited in this JSON and will be created by default anyways when creating the distribution. Here you can specify the followin options:
      • allow-all: Viewers can use HTTP or HTTPS.
      • redirect-to-https: If a viewer submits an HTTP request, CloudFront returns an HTTP status code of 301 (Moved Permanently) to the viewer along with the HTTPS URL. The viewer then resubmits the request using the new URL.
      • https-only: If a viewer sends an HTTP request, CloudFront returns an HTTP status code of 403 (Forbidden).
      It is important to mention that the only way to guarantee that viewers retrieve an object that was fetched from the origin using HTTPS is never to use any other protocol to fetch the object. If you have recently changed from HTTP to HTTPS, we recommend that you clear your objects' cache because cached objects are protocol agnostic. That means that an edge location will return an object from the cache regardless of whether the current request protocol matches the protocol used previously.
    • AllowedMethods: Here we specify the methods that you want to allow viewers to use when they access your distribution. In this case, all 7 HTTP methods are allowed, and only GET and HEAD requests will be cached, this is specified in the JSON for the CachedMethods.
    • CachedMethods: Specify only the HTTP methods that will be cached by the viewers when they access the distribution.
  • ForwardedValues: This field specifies how CloudFront handles query strings, cookies and HTTP headers. IMPORTANT: This field is deprecated and it is recommended to use a cache policy or an origin request policy instead of this field.
    • QueryString: Indicates whether you want CloudFront to forward query strings to the origin that is associated with this cache behavior and cache based on the query string parameters. It is set to false so CloudFront doesn't forward any query string parameters to the origin, and doesn't cache based on query string parameters.
    • Cookies: A complex type that specifies whether you want CloudFront to forward cookies to the origin and, if so, which ones. For more information about forwarding cookies to the origin, see How CloudFront Forwards, Caches, and Logs Cookies.
      Here we forward none so any of the cookies are forwarded by CloudFront.
    • Headers: A complex type that specifies the Headers , if any, that you want CloudFront to forward to the origin for this cache behavior. For the headers that you specify, CloudFront also caches separate versions of a specified object that is based on the header values in viewer requests.
    • QueryStringCacheKeys: A complex type that contains information about the query string parameters that you want CloudFront to use for caching for this cache behavior.
  • DefaultTTL, MinTTL and MaxTTL: Time to Live (in seconds), the default, minimum and maximun of time that objects will stay in CloudFront caches before CloudFront forwards another request to the origin to determine whether the object has been updated.
  • Comment: Any comment for the distribution.
  • Enabled: From this field, you can enable or disable the selected distribution.
  • PriceClass: The price class for the distribution. This sets the maximum price that you're willing to pay for CloudFront service. If you specify PriceClass_All, CloudFront responds to requests for your objects from all CloudFront edge locations.
    If you specify a price class other than PriceClass_All, CloudFront serves your objects from the CloudFront edge location that has the lowest latency among the edge locations in your price class. Viewers who are in or near regions that are excluded from your specified price class may encounter slower performance.
  • ViewerCertificate: A complex type that determines the distribution's SSL/TLS configuration for communicating with viewers.
    • ACMCertificateArn: If the distribution uses Aliases (alternate domain names or CNAMEs) and the SSL/TLS certificate is stored in Certificate Manager (ACM), provide the Amazon Resource Name (ARN) of the ACM certificate. CloudFront only supports ACM certificates in the US East (N. Virginia) Region (us-east-1 ).
      If you specify an ACM certificate ARN, you must also specify values for MinimumProtocolVersion and SSLSupportMethod8.
    • SSLSupportMethod If the distribution uses Aliases (alternate domain names or CNAMEs), specify which viewers the distribution accepts HTTPS connections from.
      • sni-only: The distribution accepts HTTPS connections from only viewers that support server name indication (SNI). This is recommended. Most browsers and clients support SNI.
      • vip: The distribution accepts HTTPS connections from all viewers including those that don't support SNI. This is not recommended, and results in additional monthly charges from CloudFront.
      • static-ip: This specifies if you have a use case that requires static IP addresses for the distribution.
        • This is set to sni-only This means that CloudFront will only use Server Name Indication (SNI) to associate a domain name with an SSL/TLS certificate. IMPORTANT: SNI is an extension to the Transport Layer Security (TLS) protocol that allows a server to present multiple digital certificates on the same IP address and port number, and thus eliminates the need for a dedicated IP address for each SSL certificate. When a browser connects to a website over HTTPS, it sends the hostname of the website in the SNI field of the SSL/TLS handshake. The server then uses that hostname to determine which certificate to present to the browser for the secure connection. So by using sni-only, we are telling CloudFront to only serve HTTPS requests for the domain with a certificate that matches the domain name specified in the SNI field of the SSL/TLS handshake. This is a more cost-effective option as it doesn't require a dedicated IP address for each SSL certificate.
    • MinimumProtocolVersion: If the distribution uses Aliases (alternate domain names or CNAMEs), specify the security policy that you want CloudFront to use for HTTPS connections with viewers. The security policy determines two settings:
      • The minimum SSL/TLS protocol that CloudFront can use to communicate with viewers.
      • The ciphers that CloudFront can use to encrypt the content that it returns to viewers.
  • IsIPV6Enabled: If you want CloudFront to respond to IPv6 DNS requests with an IPv6 address for your distribution, specify true . If you specify false , CloudFront responds to IPv6 DNS requests with the DNS response code NOERROR and with no IP addresses. This allows viewers to submit a second request, for an IPv4 address for your distribution.

If you're using an Route 53 Amazon Web Services Integration alias resource record set to route traffic to your CloudFront distribution, you need to create a second alias resource record set when both of the following are true:

  • You enable IPv6 for the distribution.
  • You're using alternate domain names in the URLs for your objects.

If you created a CNAME resource record set, either with Route 53 Amazon Web Services Integration or with another DNS service, you don't need to make any changes. A CNAME record will route traffic to your distribution regardless of the IP address format of the viewer request.

Minification refers to the process of removing unnecessary characters from website resources, such as HTML, CSS, and JavaScript files, in order to reduce their file size and improve website performance.
During this process, we can eliminate reduntant code such as comments, white space, and formatting, as well as renaming variables amd function names to shorten them.
Some common techniques used in minifying website resources are:

  • Removing unnecessary whitespace
  • Removing comments from HTML, CSS, and JavaScript code.
  • hortening variable and function names to reduce their length
  • emoving unused code, such as CSS rules that are not being applied or JavaScript functions that are not being used.
  • Combining multiple CSS or JavaScript files into a single file to reduce the number of requests made to the server.
  • Using shorthand syntax for CSS properties, such as using "margin: 0" instead of "margin-top: 0; margin-right: 0; margin-bottom: 0; margin-left: 0;"
  • Removing redundant code, such as duplicate CSS rules or unused media queries.
  • Minimizing the use of images or optimizing images to reduce their file size.

Different minification techniques can be used depending on the type of file being minified and the specific requirements of the website or application. However, it's important to note that not all code can be safely minified, and some code may require different or specialized minification techniques to avoid errors or loss of functionality.
Minifying website resources can help reduce page load times, improve website performance, and reduce server load, especially on slower connections or devices. It can also potentially improve search engine rankings and visibility.

Advantages of Minification:

  • Improved website performance: Smaller file sizes mean faster load times and reduced bandwidth usage, which can significantly improve website performance, especially on slower connections or devices.
  • Reduced server load: Smaller file sizes also mean less strain on the server, which can result in faster page rendering and lower server costs.
    Server strain refers to the load or stress placed on a server as a result of processing requests and delivering content to clients. When a server receives a request from a client, it must process the request, retrieve the requested content, and then send it back to the client over the network. This process can be resource-intensive, especially for high-traffic websites or complex web applications that require significant processing power and memory.
  • Better SEO (Search Engine Optimization): Search engines consider page load time as a ranking factor, so faster loading pages can improve a website's search engine ranking and visibility.
    SEO refers to the process of improving the visibility and ranking of a website or webpage in search engine results pages (SERPs). The objective of this is to increase organic (non-paid) traffic to a website by optimizing various on-page and off-page factors that search engines use to determine the relevance and authority of a website or webpage. These factors include website content, keywords, backlinks, website structure, page speed, mobile responsiveness, and more. By optimizing these factors, a website can improve its search engine ranking and visibility, which can result in more traffic and potential customers.

Although, Minifying website resource has great advantages, it also has disadvantages when using minifying techniques because some minification techniques, such as renaming variables or function names, can create issues with certain browsers or devices that do not support these changes. This can lead to unexpected behavior or errors on the website, which can negatively impact user experience.

Disadvantages of Minification:

  • Reduced readability: Minified code can be more difficult to read and understand, especially for developers who are not familiar with the specific minification tool used.
  • Debugging difficulties: When an error occurs in minified code, it can be challenging to identify the specific line of code that is causing the issue, making debugging more difficult.
  • Potential loss of functionality: In some cases, minification can remove important code that is needed for website functionality, leading to issues with the website's usability or performance.

As mention, another disadvantage of minifying website resources is the potential for compatibility issues with certain browsers or devices. Some minification techniques, such as renaming variables or function names, can create issues with certain browsers or devices that do not support these changes. This can lead to unexpected behavior or errors on the website, which can negatively impact user experience.
Additionally, some minification tools may not be able to distinguish between essential and non-essential code, and may inadvertently remove code that is needed for specific website features or functionality.

It is important to thoroughly test the website after minification and to ensure that it remains fully functional across all target browsers and devices. In some cases, it may be necessary to use different minification techniques or to disable certain minification features in order to ensure compatibility and maintain website functionality.

If-else does not handle exceptions If an exception occurs within an if-else block, it will not be caught and handled by the block itself, but will instead propagate up the call stack until it is caught by another try-except block or until it causes the program to terminate.

Try-except is good when working with external resources or APIs that may raise exceptions, as it allows you to handle those exceptions in a controlled manner and recover from errors without crashing your program.

According to Carol Dweck, there are two types of mindsets that people adopt: a growth mindset and a fixed mindset.

  • Growth Mindset: A growth mindset is a belief that one's abilities and intelligence can be developed and improved through hard work, dedication, and persistence. People with a growth mindset embrace challenges, persist in the face of setbacks, and are not afraid to fail, as they see it as an opportunity to learn and grow. They also celebrate the success of others and see it as a source of inspiration and motivation for their own growth and development.
  • Fixed Mindset: A fixed mindset is a belief that one's abilities, talents, and intelligence are fixed and cannot be changed or improved upon. People with a fixed mindset often avoid challenges, give up easily when faced with obstacles, and fear failure, as it confirms their belief in their own limitations. They may also feel threatened by the success of others, as it undermines their own sense of competence.

People with a growth mindset are more likely to achieve their full potential and succeed in life, as they are willing to take risks, learn from their mistakes, and put in the effort needed to improve their skills and abilities. In contrast, people with a fixed mindset may limit their own potential and miss out on opportunities for growth and development.
